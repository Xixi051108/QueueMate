# Vue3 全角色前端完成记录

## 1. 本轮目标

在不修改后端业务逻辑和接口的前提下，把 QueueMate 从用户端第一版扩展为访客、普通用户、商家和管理员都可使用的完整 Web 前端。

## 2. 功能覆盖

### 访客与普通用户

- 新增普通用户注册页，复用后端用户名、密码、显示名称和手机号校验边界。
- 登录后按角色进入对应工作区，原目标路由继续保留。
- 地点类别改为后端真实枚举 `TEA_SHOP / STUDY_ROOM / BADMINTON_COURT`。
- 我的预约支持状态筛选、取消原因、消费码有效期和复制。
- 我的排队支持按日期、地点和状态查询历史号码。
- 钱包流水根据 `balanceBefore` 与 `balanceAfter` 判断收支方向，不能直接把始终为正数的 `amount` 当成入账。

### 商家与管理员场所运营

- 地点创建、修改、启用和停用。
- 预约时段创建、日期/状态筛选、打开和关闭。
- 查看当天或指定日期排队进度，对 `WAITING` 叫号，对 `CALLED` 完成或过号。
- 输入消费码核销收费预约。
- 按日期范围查看预约与排队的小时热度。
- 商家前端只展示自己名下地点，后端继续执行最终资源归属校验。

### 管理员工具

- 管理全部地点并复用场所运营工作台。
- 查询全局钱包流水，按用户 ID 和流水类型筛选。
- 对指定用户执行正负余额调整。
- 按预约 ID 执行管理员取消。

## 3. 后端接口边界如何反映在页面

前端不能伪造后端没有提供的数据：

- 后端没有商家用户列表接口，因此管理员创建地点时需要手动输入商家用户 ID。
- 后端没有全局预约列表接口，因此管理员取消预约时需要手动输入预约 ID。
- 钱包流水响应没有用户 ID，管理员需要先按用户 ID 发起筛选，页面明确提示这一点。

这些是 API 能力边界，不应该在前端通过静态假数据伪装成完整查询功能。

## 4. 设计系统落实

- 继续使用 `design-system/MASTER.md` 的城市服务台方向。
- 注册页使用“创建账号 -> 选择地点 -> 预约或取号”的服务流程作为唯一视觉签名。
- 商家运营页按真实业务链组织时段、排队、核销和统计，而不是拆成无关联的模板卡片。
- 使用服务蓝、冷灰画布、矩形票据、虚线分隔和等宽数字；没有渐变、玻璃效果、过度圆角或装饰动画。
- 增加跳到主要内容的键盘入口、持久表单标签、加载按钮、错误反馈和 `prefers-reduced-motion` 支持。
