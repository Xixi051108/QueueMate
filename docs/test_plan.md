# QueueMate 测试计划

## 1. 测试目标

QueueMate 的测试目标不是验证简单页面展示，而是围绕业务规则、权限控制、模拟余额支付、状态流转和并发一致性来体现测开价值。

重点验证：

- JWT 鉴权是否可靠
- RBAC 权限边界是否清晰
- 预约是否存在超卖、重复预约、一致性异常
- 模拟余额支付是否存在重复扣款、余额不足、退款异常
- 排队状态流转是否符合预期
- Web UI 主流程是否可自动化回归
- CI 是否能承接基础质量门禁

## 2. 测试范围

### 2.1 范围内

- 后端接口功能测试

## 5. 核心测试主题

## 5.1 鉴权测试

- 未登录访问受保护接口，应返回未授权
- 携带伪造 token，应返回未授权
- 使用已禁用账号登录，应返回失败
- 使用错误密码登录，应返回失败

## 5.2 权限测试

- `USER` 不可创建地点
- 匿名用户可查询地点列表和详情
- `MERCHANT` 创建地点时归属自动绑定当前商家
- `ADMIN` 可为有效商家创建地点
- `USER` 不可叫号
- `MERCHANT` 不可操作其他商家的地点
- `MERCHANT` 不可查看其他商家受限统计
- `ADMIN` 可全局操作

## 5.3 预约测试

- 匿名用户可按日期范围和状态查询地点时段
- 商家可为自己的可预约地点创建时段
- 商家不可为其他商家的地点创建或关闭时段
- 已停用或未启用预约的地点不可创建时段
- 过去日期、非法时间范围、非正容量和负数价格被拒绝
- 相同地点、日期和时间范围的重复时段被拒绝
- 商家或管理员可关闭时段
- 地点不可用时，已关闭时段不可重新打开
- 用户成功预约开放且未满的时段
- 同一用户重复预约同一时段被拒绝
- 已满时段预约失败
- 已关闭时段预约失败
- 取消预约成功后名额回补
- 重复取消同一预约被拒绝

## 5.4 排队状态流转测试

- 用户成功取号
- 商家可对 `WAITING` 执行叫号
- 商家可对 `CALLED` 执行完成
- 商家可对 `CALLED` 执行过号
- `WAITING -> COMPLETED` 必须失败
- `COMPLETED -> CALLED` 必须失败

## 5.5 模拟支付测试

- 用户可模拟充值成功
- 用户可查看钱包余额
- 用户可查看自己的钱包流水
- 余额不足时收费预约失败
- 预约失败时不得产生成功扣款流水
- 收费预约成功后余额减少，流水新增 `PAYMENT`
- 收费预约成功后生成金额一致、状态为 `AVAILABLE` 的唯一消费码
- 取消已支付预约后余额增加，流水新增 `REFUND`
- 退款成功后消费码同步变为 `VOID`
- 重复取消预约不得重复退款
- 普通用户不可查看其他用户钱包流水
- 管理员可查看全局钱包流水

## 5.6 并发一致性测试

- 给某时段设置容量 `N`
- 以远大于 `N` 的并发数提交预约
- 最终成功预约数不得超过 `N`
- `booking_slots.reserved_count` 必须与有效预约数一致
- 不允许出现重复预约脏数据
- 并发取消同一预约时只能有一个请求成功，名额只回补一次
- 钱包余额不得被扣成负数
- 同一预约不得出现多条成功扣款流水
- 同一收费预约不得生成多张消费凭证
- 并发核销同一消费码时只能有一个请求成功

## 5.7 消费码核销测试

- 用户只能查看自己预约的完整消费码
- 商家预约列表不返回完整消费码
- 所属商家可在核销窗口内成功核销，预约变为 `FULFILLED`
- 商家不可核销其他地点的消费码
- 不存在、已作废、已过期消费码核销失败
- 同一消费码重复核销失败
- 核销窗口开始前或结束后核销失败
- 已核销预约不可取消或退款
- 退款与核销并发时只能有一个状态路径成功
- 日志和错误响应不得泄露完整消费码

## 5.8 统计测试

- 指定地点统计接口返回小时维度数据
- 统计结果字段完整
- 无数据时返回空列表而不是异常

## 5.9 UI 自动化测试

- 用户登录后可浏览地点列表
- 用户可完成模拟充值
- 用户可进入地点详情并提交预约
- 用户可查看预约扣款结果和钱包流水
- 用户可查看并出示付费预约消费码
- 用户可在我的预约中取消预约
- 用户取消后可看到退款结果
- 商家可输入或扫描消费码完成核销
- 商家登录后可查看本店排队并执行叫号
- 页面应对异常提示有可断言反馈

- `WalletServiceTest`：5 个，覆盖充值、付费扣款、余额不足、冻结钱包和退款。

- 当前 Maven 全量共执行 94 个测试，结果为 0 失败、0 错误、0 跳过。

真实 HTTP + MySQL 并发验证：

- 创建容量为 3 的临时免费时段，12 个不同用户同时预约。
- 最终 3 个请求成功、9 个请求返回 `409/BOOKING_SLOT_FULL`。
- 数据库中 `reserved_count=3`，有效 `BOOKED` 预约数为 3，重复用户时段组合为 0。
- 同一预约并发取消两次时，1 次成功、1 次返回 `409/BOOKING_STATUS_INVALID`。
- 取消释放名额后，先前失败用户可成功预约，最终容量和有效预约数仍一致。
- 并发验证产生的预约、时段、钱包和用户临时数据已全部清理。

钱包与消费码真实 HTTP + MySQL 验证：

- 新用户钱包初始余额为 0，模拟充值 100 后可查询充值流水。
- 收费预约 20 成功后余额变为 80，并生成 `PAID` 预约、`PAYMENT` 流水和 `AVAILABLE` 消费码。
- 商家核销后预约为 `FULFILLED`、消费码为 `REDEEMED`；重复核销和核销后取消均返回 409。
- 第二笔收费预约取消后预约为 `CANCELLED/REFUNDED`、消费码为 `VOID`，钱包退回到 80。
- 容量 3 的收费时段由 6 个已充值用户并发预约，3 个成功、3 个 409；成功者余额 30、失败者余额 50。
- 数据库最终为 3 条支付流水、3 张消费凭证、`reserved_count=3`，同一消费码并发核销结果为 200/409。
- 本轮真实回归的临时用户、钱包、流水、预约、消费码和时段已全部清理。

Postman 已覆盖认证、地点、预约时段、钱包查询、充值、流水、免费预约、付费预约、取消退款和作废消费码。当前共 32 个请求且 JSON 可解析；只有实际运行 Postman Runner 或 Newman 后才能标记接口集合通过。
