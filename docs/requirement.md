# QueueMate 需求说明

## 1. 项目背景

QueueMate 是一个用于练习全链路测开能力的生活排队与预约平台。项目以模拟线下生活场景为载体，覆盖账号鉴权、角色权限、预约容量控制、模拟余额支付、现场叫号、状态流转、统计分析等核心业务。

项目重点不在于接入真实第三方服务，而在于构建一个适合做接口测试、权限测试、并发测试、UI 自动化测试和 CI 演示的完整样例。

## 2. 项目目标

- 完成一个可演示的 Web UI 项目
- 支撑 Spring Boot + Vue3 + MySQL 的端到端开发
- 为 Postman、JMeter、Playwright、GitHub Actions 提供明确测试对象
- 突出测开价值，包括业务规则验证、异常场景验证和自动化回归能力

## 3. 角色定义

### 3.1 USER

- 注册、登录平台
- 浏览地点和可预约时段
- 提交预约、取消预约
- 使用站内余额预付收费时段的消费金额
- 查看并出示付费预约的消费码
- 查看自己的余额和钱包流水
- 在线查看自己的预约记录
- 对支持现场取号的地点进行取号

### 3.2 MERCHANT

- 登录商家账号
- 管理自己名下地点
- 配置地点预约时段和容量
- 查看本店预约与现场排队情况
- 查看本店预约支付状态
- 核销自己地点的预约消费码
- 对当前号码进行叫号、完成、过号操作

### 3.3 ADMIN

- 拥有全局管理权限
- 管理所有地点和账号状态
- 查看全局统计数据
- 处理异常或违规数据

## 4. 业务对象

- 用户 `User`
- 地点 `Venue`
- 预约时段 `BookingSlot`
- 预约记录 `Booking`
- 预约消费凭证 `BookingVoucher`
- 钱包 `Wallet`
- 钱包流水 `WalletTransaction`
- 现场号码 `QueueTicket`

## 5. MVP 功能范围

### 5.1 用户与鉴权

- 用户注册
- 用户登录
- JWT 鉴权
- 获取当前登录用户信息

### 5.2 角色权限

- 基于角色进行资源访问控制
- 普通用户、商家、管理员权限边界清晰
- 商家只能管理自己名下资源

### 5.3 地点管理

- 支持奶茶店、自习室、羽毛球场等模拟地点
- 支持地点新增、编辑、启停用
- 支持按类型、状态查看地点

### 5.4 时间段预约

- 使用固定时段槽位模型
- 每个时段有容量上限
- 支持公开查询地点时段
- 地点所属商家或管理员可创建、打开和关闭时段
- 用户可预约和取消预约
- 限制同一用户同时持有多笔同一时段的有效预约；取消后允许重新预约
- 支持免费预约和收费预约
- 收费预约使用站内钱包预付消费金额并生成消费码

### 5.5 模拟余额支付

- 每个用户拥有一个站内余额账户
- 用户可进行模拟充值，系统直接增加余额
- 收费时段的价格表示预付消费金额，不是额外预约手续费
- 收费预约需要先完成余额扣款
- 余额不足时预约失败，前端提供前往充值的恢复入口
- 从预约进入钱包充值成功后，返回原地点预约页面重新确认
- 取消已支付预约时按规则退回余额
- 所有充值、扣款、退款都必须记录钱包流水
- 不接入真实微信、支付宝、银行卡或第三方支付网关

### 5.6 预约消费码

- 收费预约支付成功后生成一张与预约一对一的消费凭证
- 消费码代表该预约已预付的消费金额，可用于到店抵扣或兑换对应服务
- 用户可在“我的预约”中查看消费码、预付金额和核销状态
- 地点所属商家或管理员可核销消费码
- 核销成功后预约状态变为 `FULFILLED`，消费码不可再次使用
- 未核销预约退款时消费码必须同时作废
- 超过使用期限且未核销时，消费码可标记为 `EXPIRED`，预约变为 `NO_SHOW`

### 5.7 现场取号

- 用户可以取号
- 商家可以叫号、完成、过号
- 支持查看当前叫号进度

### 5.8 繁忙时段统计

- 根据预约记录和排队记录统计地点繁忙时段
- 支持作为测试和展示用的简化统计接口

## 6. 非功能和边界约束

- 不接真实地图 API
- 不接短信验证码
- 不接微信登录
- 不接真实支付平台，只做站内余额和模拟支付
- 所有地点、时段、商家和用户数据均使用模拟数据或本地数据库数据
- 优先保证可实现、可测试、可演示

## 7. 关键业务规则

### 7.1 认证与权限

- 用户登录成功后获取 JWT
- 访问受保护接口必须携带有效 JWT
- 商家仅能操作自己名下地点、时段、预约和队列
- 管理员可全局操作

### 7.2 预约规则

- 一个预约必须绑定一个地点时段
- 时段日期不得早于创建当天，开始时间必须早于结束时间
- 时段容量必须大于 `0`，价格不得小于 `0`
- 同一地点不可创建日期、开始时间和结束时间完全相同的时段
- 只有启用且支持预约的地点可以创建或重新打开时段
- 时段预约数不可超过容量上限
- 同一用户不可同时持有多笔同一时段的 `BOOKED` 预约
- 已取消预约不计入当前已预约数
- 用户取消预约后，在时段仍开放、未开始且有余量时可以重新预约
- 已停用地点或已关闭时段不可继续预约
- 收费预约必须完成支付后才算预约成功
- 收费预约成功后必须生成金额等于 `paid_amount` 的可用消费凭证
- 预约名额增加、预约记录创建必须在同一事务内完成，任一步失败均回滚
- 取消预约只允许从 `BOOKED` 流转为 `CANCELLED`，状态更新与名额回补必须在同一事务内完成
- 免费预约成功时支付状态为 `NOT_REQUIRED`、实付金额为 `0`

### 7.3 支付规则

- 站内余额只用于项目内模拟支付
- 模拟充值无需真实支付渠道
- 收费时段的 `price` 是预付消费金额，可在核销时抵扣消费或兑换服务
- 余额扣减与预约创建必须在同一事务内完成
- 预约创建失败时不得扣款
- 同一预约不得重复扣款
- 付费预约在时段开始前且消费凭证仍为 `AVAILABLE` 时允许全额退款
- 时段开始后首版不允许用户自助取消；异常情况由管理员后续处理
- 取消已支付预约时只能退款一次
- 钱包余额不得小于 `0`
- 钱包流水金额和钱包余额变动必须保持一致

### 7.4 消费码规则

- 每个收费预约最多存在一张消费凭证，免费预约不生成消费码
- 消费码必须使用不可预测的随机值并保持全局唯一，不得由预约 ID 直接推导
- 消费凭证状态为 `AVAILABLE` / `REDEEMED` / `VOID` / `EXPIRED`
- 商家只能核销自己地点的消费码，管理员可全局核销
- 核销时预约必须为 `BOOKED`、支付状态必须为 `PAID`、消费凭证必须为 `AVAILABLE`
- 首版默认核销窗口为预约开始前 30 分钟至时段结束时间
- 核销成功必须原子更新消费凭证为 `REDEEMED`、预约为 `FULFILLED` 并记录核销人和核销时间
- 已核销消费码不可取消、退款或再次核销
- 取消退款必须原子更新预约、退款流水、钱包余额、名额和消费凭证状态
- 接口日志和错误日志不得输出完整消费码

### 7.5 现场取号规则

- 现场号码按地点和日期生成递增序号
- 号码状态流转固定为 `WAITING -> CALLED -> COMPLETED/MISSED`
- 非法状态跳转必须被拒绝
- 已停用地点不可继续取号

### 7.6 统计规则

- 首版统计按地点维度输出
- 首版只统计固定时间范围内的高峰时段
- 首版使用聚合查询，不单独维护统计表

## 8. 用户故事与验收标准

### 8.1 USER 视角

故事：作为普通用户，我希望注册并登录平台，以便查看地点并进行预约。

验收标准：

- 用户可使用用户名和密码注册
- 用户可登录成功并拿到 token
- 用户登录后可查看地点列表和可预约时段
- 用户可成功预约未满的时段
- 免费预约成功后可在“我的预约”中查询
- 收费预约会在钱包模块上线后扣减站内余额、生成支付流水和消费码
- 用户可查看付费预约对应的预付金额和消费码状态
- 用户取消预约后可在我的预约中看到状态变化
- 用户取消已支付预约后余额会按规则退回

故事：作为普通用户，我希望避免误操作或重复预约。

验收标准：

- 同一用户已有 `BOOKED` 预约时，重复预约同一时段返回失败
- 原预约已取消时，允许重新预约同一时段并生成新的预约记录
- 当时段已满时，预约请求返回失败
- 当余额不足时，收费预约返回失败，用户可前往钱包充值并返回原预约页面
- 重复提交同一收费预约不得重复扣款
- 未登录用户调用受保护接口时返回未授权
- 高并发请求同一时段时，成功预约数不得超过时段容量
- 取消成功后名额回补一次，重复取消不得重复回补

### 8.2 MERCHANT 视角

故事：作为商家，我希望管理自己的地点和预约资源。

验收标准：

- 商家可新增和编辑自己名下地点
- 商家可配置地点预约时段与容量
- 商家可查看自己地点的预约记录
- 商家可查看预约支付状态，但不可直接调整用户余额
- 商家可在有效核销窗口内核销自己地点的消费码
- 重复核销、跨店核销和已退款消费码核销必须失败
- 商家不可修改其他商家的地点和排队数据

故事：作为商家，我希望处理现场排队。

验收标准：

- 商家可查看当前排队号码
- 商家可执行叫号、完成、过号操作
- 非法状态流转会被拒绝并返回明确错误

### 8.3 ADMIN 视角

故事：作为管理员，我希望从全局管理平台资源。

验收标准：

- 管理员可查看和管理所有地点
- 管理员可启停用异常地点或账号
- 管理员可查看钱包流水，必要时可做模拟余额调整
- 管理员可查看统计接口结果

## 9. 重点测试价值

- 鉴权测试：登录态、伪造 token、过期 token
- 权限测试：不同角色访问不同资源
- 状态流转测试：预约取消、叫号完成、过号路径
- 并发测试：同一时段高并发预约防超卖
- 支付测试：余额不足、重复扣款、取消退款、并发扣款
- 一致性测试：取消回补名额、重复预约限制
- UI 自动化测试：用户端和商家端主流程回归
